Notes Between Deploys: An archive of ideas I would otherwise forget

Published 2026-02-05

I write code, and then, inevitably, I forget why I wrote it that way. Six months later, I’ll look at a project and wonder who made such bizarre architectural choices, only to realize that person was me. Welcome to Notes Between Deploys, my attempt to fix that.

What I built

I built a minimalist website to document my projects. The goal is simply to have a permanent record of the design decisions, the tech stack choices, and the trade-offs I made along the way. It’s essentially an external hard drive for my working memory.

It's built for me, but if you happen to find my notes useful (or mildly entertaining), that's a nice bonus. The main outcome is that this site is now live and ready to hold my thoughts before they vanish.

Highlights

  • Key learning #1: Code doesn't explain the "why." A Git commit tells me what files changed, but it doesn't capture the three hours I spent researching databases before settling on one. I realized I needed a dedicated space for the context behind the code.

  • Key learning #2: Friction is the enemy of documentation. If the process of writing a post is complicated, like dealing with a clunky CMS or configuring a build pipeline, I simply won't do it. I decided to write everything in raw HTML. No Markdown parsers, no complex editors. Just me, a few <p> tags, and my notes.

  • Key learning #3: Shipping is better than tweaking. The hardest part of building is knowing when the product is finished. I had to draw a line in the sand and just get the core structure, without the blogs for my existing projects, deployed.

Architecture Approach

I kept the architecture deliberately boring. It’s a purely static site—no heavy backend, no database to maintain, just plain HTML files served straight to the browser.

The main tradeoff is that the site is incredibly bare-bones. There’s no magic happening under the hood. If I want a search feature or a tagging system later, I’ll have to build it myself. If I were to start over, I might look into a framework that gives me a few more features out of the box, but for version 1.0, keeping it this simple was the right call.

The Branching Strategy

You might think a website consisting entirely of plain text wouldn't need a formalized Git branching strategy. You would be wrong. I implemented a three-branch system, namely dev, main, and prod, mostly to keep it as close to the industry standard as possible.

  • The dev branch: This is the drafting room. It’s where new notes and features are merged and validated before they see the light of day.

  • The main branch: This is the vault. It is the absolute, production-ready source of truth. I keep main pristine so that if the files ever get hopelessly corrupted in dev or prod, I can just clone main, redeploy, and pretend the outage never happened.

  • The prod branch: This holds the code currently running live on the internet. If you are reading this, you are looking at the prod branch.
You might notice a glaring omission: Where is the staging branch?

There isn't one. The reality is, I am just publishing text. I don't have complex user authentication, fragile APIs, or special features to test. Thorough testing and multiple deployment iterations to hunt down faults are completely unnecessary here. If I make a typo in a paragraph, I hope I don't get judged too harshly.

Results

Performance is excellent, mostly because serving plain HTML text is about as fast as the internet gets. Reliability is currently 100%. But the real impact is personal: I can now safely flush the intricate details of past projects out of my brain, knowing they are safely archived right here.
